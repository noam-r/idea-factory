# Compile

## Purpose of Compile

Compile turns the concept into a readable, navigable artifact.

By the time a project reaches this stage, the idea should already be:

- clarified,
- structured,
- and evaluated.

Compile should not be used to rescue weak thinking. Its purpose is to express the current concept clearly.

## What Compile is working with

Compile works from:

- the structured page model,
- the clarified concept definition,
- the Evaluation output,
- the current project goal,
- and the intended audience.

It should treat these as constraints, not suggestions to ignore.

## What the LLM should do in Compile

In Compile mode, the LLM should behave like a disciplined editor.

It should:

- turn the page model into draft pages,
- write clearly and economically,
- preserve user intent and concept meaning,
- keep uncertainty visible,
- and ensure the artifact is coherent as a whole.

This is where the concept site begins to feel concrete.

## What the LLM should not do in Compile

Compile should not:

- beautify the concept into something stronger than it is,
- override protected evaluation outputs,
- erase limitations,
- force generic product language,
- or treat readability as permission to distort the recommendation.

The artifact should become clearer, not more flattering.

## What a good Compile result looks like

A good Compile result should usually produce:

- a strong Overview page,
- clear supporting pages,
- consistent terminology,
- visible Evaluation content,
- and a readable but honest concept artifact.

The goal is a first serious draft, not necessarily a final public site.

## What the user should do in Compile

The user should be able to steer the artifact through prompt-based edits.

Typical user instructions here may include:

- make this page shorter,
- simplify the wording,
- add one concrete example,
- make the explanation more suitable for a trusted review audience,
- or tighten the Overview.

Direct free-form editing is not the model. Compile remains governed.

## What remains protected during Compile

Even in Compile mode, protected layers stay protected.

The user may not directly overwrite:

- the Evaluation conclusion,
- the recommendation,
- readiness,
- convergence,
- or milestone history.

If the user disputes the Evaluation, the proper route is challenge and reevaluation, not direct rewrite.

## What blocks Compile

Compile should stop or send the project back when:

- the concept is still too unstable,
- the structure is not actually coherent,
- the evaluation is incomplete,
- or the system would be polishing over unresolved conceptual weakness.

A good artifact cannot be built on top of a concept that still has no stable center.

## Compile and audience fit

Compile should adapt the wording to the current audience, but not the truth of the concept.

For example, it may make the artifact more suitable for:

- personal review,
- trusted collaborators,
- or broader external readers.

But that adaptation must not erase uncertainty or weaken the integrity of the evaluation.

## Compile and completion

In many projects, Compile is the point where the project becomes **complete enough for now**.

If the user's goal is to create a clear concept artifact for review or later continuation, a strong draft may be enough.

The system should not push toward publication automatically just because a draft now exists.

## Compile and session efficiency

Compile is one of the stages most vulnerable to diminishing returns.

It is easy to keep rewriting wording while creating little additional value.

The Progress Efficiency Model should intervene when the session shifts from:

- improving clarity,

to

- polishing the same concept without meaningful gain.

At that point, the system should recommend stopping, pausing, or moving to Publish only if the current goal requires it.

## Summary

Compile is the stage where Idea Factory turns a judged concept into a usable artifact.

Its job is to create clarity and navigability without upgrading the idea through presentation alone.
