# Publish

## Purpose of Publish

Publish is the stage where the system decides whether a concept artifact is fit to be exposed to a chosen audience.

This stage is not about generating enthusiasm. It is about controlled release.

The main question is:

- is this artifact clear enough, honest enough, and stable enough for the audience the user has chosen?

## What Publish is working with

Publish works from:

- the compiled artifact,
- the current Evaluation,
- the stated audience,
- and the desired visibility mode.

It assumes that the concept has already been clarified, structured, and evaluated.

## What the LLM should do in Publish

In Publish mode, the LLM should be conservative.

It should:

- check whether the artifact fits the intended audience,
- preserve known limitations,
- state whether publication is advisable,
- create a publication snapshot,
- and record what was exposed and why.

This is the stage where the system protects against premature external exposure.

## What the LLM should not do in Publish

Publish should not:

- encourage publication for its own sake,
- remove warnings to improve optics,
- make the artifact sound more certain than it is,
- or treat any publication as a success just because it is possible.

A project can be valuable and still not be ready for publication.

## Visibility modes

Phase I should support publication choices such as:

- public,
- trusted group,
- private link,
- or internal only.

The system should help the user match the artifact to the right exposure level.

A concept may be fit for trusted review while still being unfit for broad public release.

## What the user should do in Publish

The user decides:

- whether to publish,
- who the audience is,
- and what visibility mode to use.

The user may choose to publish despite visible weaknesses, but the system should preserve those weaknesses rather than hide them.

## What Publish should produce

A successful Publish stage should produce:

- a publication recommendation,
- a recorded audience and visibility choice,
- a publication snapshot,
- and a clear record of known limitations.

This creates a durable account of what version of the concept was actually shared.

## What blocks Publish

Publish should refuse or advise against publication when:

- the artifact is still materially unclear,
- the concept is not stable enough for the chosen audience,
- the Evaluation exposes weaknesses that make broad publication misleading,
- or the user is trying to publish a concept that the integrity layer should not support.

The system should say so plainly.

## Publish and project completion

Publish is one valid completion path, but not the only one.

A project may legitimately end as:

- complete enough for now,
- paused,
- stopped,
- or ready for trusted review without broader publication.

Publish should not be treated as the default finish line for every project.

## Publish and trust

The trustworthiness of the artifact depends on the system preserving:

- evaluation integrity,
- recommendation integrity,
- and publication history.

If those can be silently rewritten, the publication loses much of its value.

That is why Publish is closely tied to the protected layers of the system.

## Publish and session efficiency

Publish should usually be a short stage.

By the time a project reaches Publish, the main question is not whether the concept exists, but whether this artifact should be exposed now.

If Publish becomes a long session, that often means the real issue lies earlier:

- incomplete compilation,
- unresolved evaluation,
- or a mismatch between artifact quality and audience ambition.

The right answer is often to step back rather than keep pushing.

## Summary

Publish is the stage where Idea Factory helps the user expose a concept intentionally and honestly.

Its job is to preserve audience fit, visible limitations, and publication history rather than treat exposure itself as the goal.
