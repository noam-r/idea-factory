# Structure

## Purpose of Structure

Structure turns a clarified idea into a navigable concept model.

At this stage, the system should stop asking whether the idea exists at all and start organizing what has already been clarified into a form that can later become a small concept site or idea brief.

The goal is not polish. The goal is shape.

## What Structure is working with

Structure begins after the project has enough stability that the system can state, with reasonable confidence:

- what the concept is,
- who it is for,
- what problem it addresses,
- how it works at a concept level,
- and what is outside its scope.

If that foundation is missing, Structure should not compensate for it.

## What the LLM should do in Structure

In Structure mode, the LLM should organize.

It should:

- propose a small set of core pages or sections,
- explain why each page exists,
- identify overlap or redundancy,
- surface assumptions, questions, and risks,
- and produce the first coherent information architecture for the concept.

A good structure should make the idea easier to understand, not merely longer.

## What the LLM should not do in Structure

Structure should not:

- create page sprawl,
- invent structure to hide conceptual weakness,
- produce publication-grade prose too early,
- or introduce sections that the concept has not earned.

A weak concept with a beautiful outline is still a weak concept.

## What a good structure looks like

In Phase I, the default artifact should remain small.

A typical first structure may include pages such as:

- Overview,
- Problem,
- Core Idea,
- How It Works,
- Who It's For,
- Evaluation,
- and Open Questions.

Optional pages should only appear if the concept clearly needs them.

## What the user should do in Structure

The user is still the concept owner.

In this stage, the user should:

- confirm or reject the proposed page structure,
- rename pages when necessary,
- remove pages that feel artificial,
- and say when the structure does not yet reflect the actual concept.

The user should not have to design information architecture from scratch, but should always be able to challenge the system's proposal.

## What Structure should produce

A successful Structure stage should produce:

- a first page model,
- a concept map,
- a list of important assumptions,
- a list of open questions,
- and the first durable shape of the future artifact.

This is the point where the project begins to feel like a concept artifact rather than an extended conversation.

## How Structure interacts with Evaluation

Structure should prepare the concept for evaluation.

That means the structure needs to make the following visible enough to judge:

- what the concept claims,
- how it is supposed to work,
- what depends on assumptions,
- and where the main uncertainties still live.

If the structure hides those things, the next stage will be weak.

## What blocks Structure

Structure should stop or send the project back to Clarify when:

- the proposed pages are compensating for conceptual confusion,
- two or more pages are trying to carry the same unresolved idea,
- the concept still has no stable mechanism,
- or the user and system cannot agree on what the artifact is organizing.

The right response is not to add more sections. The right response is to restore clarity.

## Structure and linked projects

Structure is one of the stages where divergence becomes highly visible.

If the system naturally generates two different structures for what appear to be two different concepts, it should say so plainly.

That is a sign that the project may need to:

- narrow the current concept,
- or create a new linked project.

The system should not blend two architectures into one confusing artifact.

## Structure and session efficiency

Structure should usually create visible progress fairly quickly.

If many turns happen in Structure without producing a stable page model, that is often a sign that:

- Clarify was incomplete,
- the concept is still shifting,
- or the session is producing motion without enough conceptual gain.

In those cases, the Progress Efficiency Model should push the session toward a decision rather than endless restructuring.

## Summary

Structure is the stage where the idea becomes organized enough to stand on its own.

Its job is to give the concept a small, coherent, navigable architecture without pretending that organization alone makes the idea stronger than it is.
