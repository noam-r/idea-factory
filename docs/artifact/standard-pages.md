# Standard Pages

Idea Factory Phase I uses a **small standard page set** for its concept artifacts.

These pages are not meant to be a rigid template for every idea. They are meant to provide a reliable backbone that supports explanation and evaluation.

## Core pages

The default core set is:

1. Overview
2. Problem
3. Core Idea
4. How It Works
5. Who It’s For
6. Evaluation
7. Open Questions

This is a strong Phase I default because it stays small while still covering the essential needs of a concept artifact.

## Overview

The landing page.

It should answer quickly:

- what is this?
- why does it matter?
- who is it for?
- what stage is it in?

A good Overview page should include a short concept definition and links to deeper pages.

## Problem

This page explains:

- what problem, frustration, or opportunity the idea addresses
- why the current state is not enough
- why the issue deserves attention

Weak concepts often fail here. A concept that does not clearly state the problem usually becomes vague everywhere else.

## Core Idea

This page explains the central concept in plain language.

It should answer:

- what is the main idea?
- what makes it distinct from the obvious default?
- what is the simplest correct way to frame it?

## How It Works

This page explains the mechanism at the appropriate level of abstraction.

It should not default to technical implementation. Instead, it should explain:

- how the concept operates
- what its main parts are
- what logic connects those parts

## Who It’s For

This page clarifies audience and use context.

It should answer:

- who benefits?
- in what context?
- for whom is this not intended?

Even when some of this appears in the Overview, it is usually useful to keep it explicit.

## Evaluation

This is one of the most important pages in the artifact.

It should contain:

- assessment
- open risks
- assumptions
- recommendation
- uncertainty notes

The page should stay neutral. It should not assume the idea is worth pursuing. It should show the current evaluation honestly.

## Open Questions

This page preserves what is unresolved.

It should usually distinguish between:

- blocking questions
- important questions
- optional later questions

This keeps the artifact honest and gives future sessions a clean place to resume work.

## Optional pages

Depending on the concept, the system may also add:

- FAQ
- Glossary
- Examples or Scenarios
- Alternatives or Comparison
- Roadmap or Next Steps
- Governance Considerations
- Technical Direction later

These should only appear when the concept genuinely needs them. The system should avoid page bloat.

## Type-aware variation

### Product concepts

Often benefit from added emphasis on:

- use cases
- user scenarios
- differentiation
- defensibility
- next-step validation

### Framework concepts

Often benefit from added emphasis on:

- principles
- components
- examples
- boundaries
- glossary

### Policy concepts

Often benefit from added emphasis on:

- stakeholders
- tradeoffs
- governance logic
- implementation implications

## Why the page set stays small

A small page set is important because the product is trying to create clarity, not document sprawl.

The standard pages should make the concept easier to understand, not heavier to maintain.
