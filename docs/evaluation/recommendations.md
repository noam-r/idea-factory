# Recommendation Model

## Purpose

The recommendation model translates evaluation into action.

Once a concept has been assessed, the system should not stop at description. It should state, plainly, what it thinks should happen next.

This recommendation is not a command. The user still decides what to do. But the system must be willing to make a real recommendation rather than hiding behind neutral analysis.

## What recommendations are for

The recommendation model exists to answer a practical question:

**Given what is currently known, what is the most responsible next move?**

That next move may be:

- continue,
- continue carefully,
- revise,
- pause,
- stop,
- or move the strongest emerging direction into a new linked project.

The system should not assume that every project deserves continuation.

## Recommendation classes

Phase I uses six core recommendation classes.

### Proceed

The concept is strong enough to justify continued work in its current direction.

This usually means:

- the concept is clear enough,
- the mechanism is coherent enough,
- the value case is meaningful,
- and no major contradiction blocks continuation.

Proceed does not mean the concept is proven. It means it is strong enough to deserve more effort.

### Proceed with caution

The concept has enough promise to continue, but major weaknesses or uncertainties remain.

This is appropriate when:

- the idea has real potential,
- but the system sees important fragilities,
- and the user should continue with those weaknesses in full view.

This is a disciplined continuation state, not a soft compliment.

### Revise

The current concept is not strong enough in its present form, but may become stronger if it is narrowed, reframed, clarified, or otherwise changed.

This is likely to be a common recommendation in Idea Factory.

Revise means:

- do not keep polishing the current version,
- change something real first.

### Start a new linked project

The strongest idea that emerged is no longer the current project as framed.

This recommendation is appropriate when:

- a distinct and more promising direction has emerged,
- keeping both ideas in one project would create confusion,
- or the original concept has evolved into something meaningfully different.

This protects project coherence without losing conceptual lineage.

### Pause

The idea may have some merit, but it does not currently justify more effort.

Pause is appropriate when:

- the concept is still too weak or uncertain,
- more evidence is needed,
- the user's current goal has already been met,
- or further immediate work would have low value.

Pause is not failure. It is a legitimate and often healthy outcome.

### Stop

The idea is not worth continued effort in its current form.

This may be because it is:

- too weak,
- too derivative,
- too unclear,
- too unsupported,
- too easy to replicate relative to the likely upside,
- or disallowed because of harmful intent.

Stop is a successful system outcome when justified. It prevents wasted effort.

## What drives the recommendation

The final recommendation should be informed by all evaluation criteria, but one criterion has special practical weight:

### Effort-worthiness

Effort-worthiness asks whether the idea, as currently defined, deserves more of the user's time and resources.

This is what keeps the recommendation grounded in real decision-making.

A concept may be:

- coherent,
- interesting,
- and even somewhat valuable,

but still receive **Pause** or **Stop** if it does not justify continued effort.

## Hard blockers

Some conditions should block positive recommendations even if other aspects seem promising.

Examples include:

- the idea is still too unclear to judge,
- a major contradiction remains unresolved,
- the value case is too weak,
- the mechanism is undefined in a blocking way,
- or the concept is harmful or exploitative.

In those cases, the system should not hide behind a positive-sounding recommendation.

## Recommendation and uncertainty

Every recommendation should be accompanied by:

- a short explanation of why,
- and an uncertainty note when needed.

For example:

> Recommendation: Proceed with caution  
> Reason: The concept is coherent and the value case is real, but defensibility remains weak and no external comparison has been performed.

This is the right tone. The recommendation should be clear without pretending certainty.

## Challenge and reevaluation

Because recommendation is a protected evaluative output, the user should not directly overwrite it.

However, the user must be able to challenge it.

Examples of legitimate challenge include:

- clarifying a misunderstood differentiator,
- supplying stronger evidence,
- correcting the audience or problem framing,
- or arguing that an identified risk is overstated.

The system should then reevaluate. It should not simply replace the recommendation because the user asked.

## Recommendation is not publication pressure

A positive recommendation does not mean the project must be published.

A project can validly be:

- strong enough to continue,
- but not yet strong enough to publish,
- or complete enough for private use,
- but not suitable for a public audience.

Recommendation and publication readiness must remain separate.

## Summary

The recommendation model is the action layer of evaluation.

It translates analysis into one of six next-step judgments:

- proceed,
- proceed with caution,
- revise,
- start a new linked project,
- pause,
- or stop.

Its job is not to maximize continuation. Its job is to tell the truth about what the user should do next.
